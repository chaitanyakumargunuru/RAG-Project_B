# 🔍 RAG-Project_B — Secure Multi-Tenant Document Q&A System

> A full-stack, production-ready **Retrieval-Augmented Generation (RAG)** application that lets authenticated users upload their own documents and chat with them using a locally-running LLM — with zero data leakage between users.

---

## 📌 What Is This Project?

This project implements a **complete RAG pipeline** from scratch — document ingestion, vector storage, semantic retrieval, and LLM-powered answer generation — wrapped inside a **secure, multi-user web application**.

The core idea: instead of relying on a generic LLM that knows nothing about your documents, users upload their own files (PDFs, Word docs, text files) and the system answers their questions based **only** on the content of those documents.

**Everything runs locally.** No OpenAI API. No external LLM calls. No data leaves your machine.

---

## ✨ Key Features

- 🔐 **JWT Authentication** — Secure signup/login with bcrypt password hashing and OAuth2 bearer tokens
- 🗂️ **Multi-Tenant Isolation** — Every user gets their own private ChromaDB vector collection; documents from one user are never accessible to another
- 📄 **Document Ingestion Pipeline** — Upload PDF, DOCX, or TXT files which are automatically parsed, chunked, embedded, and stored
- 🧠 **Semantic Search** — Top-K retrieval using sentence-transformer embeddings (`all-MiniLM-L6-v2`) from ChromaDB
- 🤖 **Local LLM Inference** — Answers generated by `qwen2.5:3b` running via Ollama — completely offline
- 🌐 **React Frontend** — Clean chat interface for uploading documents and asking questions
- 🗄️ **PostgreSQL Backend** — User accounts and metadata stored in a relational database via SQLAlchemy

## 🔄 How the RAG Pipeline Works

### 1. Document Ingestion (`/api/upload`)

When a user uploads a file, four steps run in sequence:

```
File Upload → load_document() → split_documents() → add_documents() → ChromaDB
```

| Step | Module | What It Does |
|-------|-------------------|---------------------------------------------|
| Parse | `loader.py` | Extracts raw text from PDF / DOCX / TXT |
| Chunk | `splitter.py` | Splits into overlapping chunks (1000 chars, 200 overlap) |
| Embed | `vectorstore.py` | Encodes chunks using `all-MiniLM-L6-v2` sentence-transformers |
| Store | `vectorstore.py` | Saves vectors into a **per-user ChromaDB collection** |

### 2. Question Answering (`/api/chat`)

When a user asks a question:

```
Question → retrieve_and_answer() → ChromaDB similarity search → Top-5 chunks → Ollama LLM → Answer + Sources
```

The retriever fetches only from **that user's collection**, constructs a context prompt, and sends it to the local `qwen2.5:3b` model via Ollama. The response includes the answer and the source document references.

## 🔐 Security Design

| Concern | Implementation |
|---|---|
| Password storage | `bcrypt` hashing via `passlib` |
| Session management | JWT tokens (HS256, 30-min expiry) |
| API protection | OAuth2 Bearer token on all `/upload` and `/chat` endpoints |
| Data isolation | Each user's vectors stored in a separate ChromaDB namespace keyed by user ID |
| File safety | Extension whitelist (`.pdf`, `.docx`, `.txt`); files prefixed with `user_{id}_` |

## 🚀 Getting Started

### Prerequisites

- Python 3.10+
- PostgreSQL running locally
- [Ollama](https://ollama.com/) installed and running

### 1. Clone and install dependencies

```bash
git clone https://github.com/chaitanyakumargunuru/RAG-Project_B.git
cd RAG-Project_B
pip install -r requirements.txt
```

### 2. Pull the LLM model via Ollama

```bash
ollama pull qwen2.5:3b
ollama serve
```

### 3. Set up PostgreSQL

Create a database named `rag_db` with user `postgres` and password `password` (or update `main.py` with your credentials).

```sql
CREATE DATABASE rag_db;
```

### 4. Run the backend

```bash
uvicorn main:app --reload
```

The API will be live at `http://localhost:8000`. Interactive docs at `http://localhost:8000/docs`.

### 5. Run the frontend

```bash
cd frontend-client
npm install
npm start
```

## 📡 API Endpoints

| Method | Endpoint | Auth Required | Description |
|---|---|---|---|
| `POST` | `/api/signup` | ❌ | Register a new user |
| `POST` | `/api/login` | ❌ | Login and receive JWT token |
| `POST` | `/api/upload` | ✅ | Upload and index a document |
| `POST` | `/api/chat` | ✅ | Ask a question about your documents |

### Example: Upload a document

```bash
curl -X POST "http://localhost:8000/api/upload" \
  -H "Authorization: Bearer <your_token>" \
  -F "file=@my_document.pdf"
```

### Example: Ask a question

```bash
curl -X POST "http://localhost:8000/api/chat" \
  -H "Authorization: Bearer <your_token>" \
  -H "Content-Type: application/json" \
  -d '{"question": "What are the main findings in the document?"}'
```

## ⚙️ Configuration (`config.py`)

All key parameters are centralised in `config.py`:

```python
CHUNK_SIZE = 1000           # Characters per text chunk
CHUNK_OVERLAP = 200         # Overlap between consecutive chunks
EMBEDDING_MODEL = "sentence-transformers/all-MiniLM-L6-v2"
OLLAMA_MODEL = "qwen2.5:3b" # Swap for any Ollama-compatible model
OLLAMA_BASE_URL = "http://localhost:11434"
TOP_K_RESULTS = 5           # Number of chunks retrieved per query
```

To use a different LLM (e.g. `llama3`, `mistral`), just change `OLLAMA_MODEL` and pull it via `ollama pull <model>`.

## 🧩 Design Decisions Worth Noting

**Why local LLM?** Complete data privacy. Documents never leave the host machine, making this suitable for sensitive or confidential content.

**Why per-user ChromaDB collections?** This enforces strict data isolation at the storage layer — even if application logic had a bug, one user's vectors cannot contaminate another's results.

**Why 1000/200 chunk settings?** This balances retrieval precision (smaller chunks = more focused matches) with context completeness (overlap prevents losing information at boundaries). These are tunable for different document types.

**Why `all-MiniLM-L6-v2`?** It is a lightweight, fast, and well-performing sentence-transformer model that runs entirely on CPU, keeping the setup dependency-free from GPU requirements.
